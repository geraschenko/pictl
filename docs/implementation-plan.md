# pictl: v1 implementation plan

Starting point for implementing agents. Read `design-decisions.md` first — it defines what pictl is and why; this document defines what to build and in what order. Each phase is independently shippable and ends with a manual verification pause. Do not start a phase until the user has signed off on the previous one. Within a phase, follow the user's working style: small coherent changes, check in after each unit, no scope expansion without asking.

## Prerequisites

- A pi binary supporting `--rpc-socket` (the tee-mode fork: `geraschenko/pi`, branch `anton/rpc-tree` — successor of `anton/pi-tee`, additionally carrying `get_entries`/`get_tree`/`navigate_tree`; spec in that repo at `docs/pi-rpc-socket-mode.md`). Build the standalone binary from the fork: `npm run build:binary` in `packages/coding-agent` produces a self-contained `dist/pi` (bun-compiled); symlink it into a PATH directory or point `PICTL_PI_BIN` at it. Verify with `pi --rpc-socket /tmp/test.sock` in a terminal — it must run interactive pi and create the socket.
- pictl resolves which pi binary to spawn from the `PICTL_PI_BIN` env var (falling back to `pi` on PATH). The resolved absolute path is recorded in `agent.json` at spawn time so dormant-agent revival uses the same binary.
- For `tail --since` and entry-cursor features (phase 3): pi must also include the `get_entries`/`get_tree` RPC commands. The user's installed fork binary already includes them; if working against a different pi build, verify before phase 3.
- For the holder's `sessions` history (phase 1): pi must broadcast a `session_changed` event (`{type, sessionFile?, sessionId}`; `sessionFile` absent for in-memory sessions) when the active session is replaced, and send the same event to each new connection immediately after its `hello` record (so a client tracks the session through one code path, no `get_state` needed). This was added to the fork as part of tee mode (handoff spec: `/tmp/pi-session-changed.md`; as of 2026-06-11 in progress — verify it has landed in the installed binary before implementing session-history tracking, and sequence that part of the holder last if needed; everything else in phase 1 is independent of it).
- Node.js + TypeScript toolchain. Key dependencies: `node-pty` (PTY allocation), `@xterm/headless` (detached screen state). RPC and session types are imported from the pi package via a **local-path dependency** on the fork checkout: `"@earendil-works/pi-coding-agent": "file:../../earendil-works/pi/packages/coding-agent"` (adjust the relative path to the actual checkout location). The fork's `dist/` must be built for the types to resolve — `npm run build` in `packages/coding-agent` does this (and `npm run build:binary` includes it). Import `RpcCommand`, `RpcResponse`, `RpcSessionState`, `SessionEntry`, `SessionTreeNode`, `RpcClient` from the package index; never hand-mirror them.

## Reference protocol facts

- Connecting to `pi.sock` yields a hello record `{"type":"hello","protocol":"pi-rpc-socket","version":1}` first, then (once the `session_changed` fork change lands) a per-connection `session_changed` event describing the current session, then broadcast events. Commands are LF-delimited JSON; responses go only to the issuing connection; `id` is optional client-side correlation.
- The command set is defined in `packages/coding-agent/src/modes/rpc/rpc-types.ts` in the pi repo and documented in `packages/coding-agent/docs/rpc.md`.
- New clients receive future events only; catch-up is explicit (`get_state`, `get_messages`, and once available `get_entries --since`).

## Phase 1: spawn, holder, registry

Deliverables:

- Step zero: a node-pty smoke test — verify `npm install` compiles the native addon on this machine (needs `make`/`g++`/`python3`) and that a PTY can be allocated, before writing any real code. This is the most likely environment failure.
- `pictl spawn [--cwd <dir>] [--id <id>] [-- <pi args...>]`
  - generates an agent id (random uuid; commands accept any unique prefix) unless given; an ambiguous prefix is an error listing the candidates (docker-style), never a guess;
  - creates `$PICTL_DIR/<id>/` (default `~/.pi/agents/`);
  - daemonizes the holder (`pictl _hold`, hidden entrypoint of the same binary) and prints the agent id.
- `pictl _hold` (hidden): allocates a PTY (default 80×24), runs `pi --rpc-socket <dir>/pi.sock <pi args...>` in it with `PI_AGENT_ID=<id>` in the environment, pipes PTY output into an @xterm/headless instance, acts as the sole writer of `agent.json` (holder pid, pi pid, cwd, pi binary path, spawn args, created-at, plus the `sessions` history: the holder connects to its own `pi.sock` as a client and appends on each `session_changed` event — the first arrives right after `hello` and describes the current session, so one code path covers startup and replacements; never poll `get_state` per event. Caveat: pi defers writing a new session file until the first assistant message, so the announced path may not exist yet; record it regardless — consumers that need the file check existence themselves. Events without `sessionFile` (in-memory sessions) are not recorded), reaps pi on exit, removes the sockets on clean shutdown. The `tty.sock` server may be stubbed until phase 2, but bind it now so the directory shape is final.
- `pictl list` — readdir + probe per agent (holder pid alive; `get_state` over `pi.sock` for status: idle/streaming/dormant/tombstoned). Human-readable table; `--json` for machines. Never revives dormant agents.
- `pictl status <agent>` — one agent, more detail (model, session file and history, cwd, pids). Never revives.
- `pictl kill <agent>` — default is polite: wait until the agent is fully quiescent (current turn finished AND queued steers/follow-ups drained: `isStreaming` false with empty pending queue), then SIGTERM pi with SIGKILL escalation, holder exits, tombstone written, directory removed. `--timeout <secs>` bounds the quiescence wait and on expiry fails _without_ killing (reports the agent is still busy); the default is to wait forever (Ctrl-C is the escape hatch). `--now` first ends the current turn via RPC `abort`, then proceeds with the graceful path. `--force` SIGKILLs pi and holder with no socket interaction (for wedged agents).
- `pictl suspend <agent>` — same quiescence wait, then stop the process but keep the directory; the agent goes dormant (see design-decisions.md "Agents outlive processes").
- `pictl resume <agent>` — revive a dormant agent: start a new holder using the pi binary path, spawn args, and cwd from `agent.json`, resuming the most recent session in the `sessions` history. No-op on a running agent.
- `pictl gc` — remove tombstoned (interrupted-kill) or corrupt agent dirs only. A dead holder is not garbage; it is a dormant agent.

Notes:

- Getting the session file path into `agent.json`: the `session_changed` event received right after `hello` carries it (`get_state` remains the fallback if the holder runs against a pi predating the event). Await the socket appearing / connecting with retry — never a fixed sleep.
- Daemonization: detach the holder fully (setsid, stdio to a log file in the agent dir). The holder must survive the spawning terminal closing.
- Revival mechanism: pi's CLI resumes a specific session via `--session <path|id>`, so revival is `pi --rpc-socket <dir>/pi.sock --session <most recent sessions entry whose file exists on disk> <recorded spawn args...>`, starting without `--session` if no entry's file exists. File existence is the only criterion.

Verification pause: spawn an agent and confirm `list` shows it running; kill -9 the holder and confirm `list` reports it dormant and `resume` revives it on the same session; round-trip `suspend`/`resume`; `purge` removes it; `gc` cleans a hand-tombstoned dir and leaves a dormant one alone.

## Phase 2: attach

Deliverables (decisions resolved 2026-06-12; rationale and spike findings in the work log):

- Shared framing module (e.g. `src/tty-protocol.ts`): binary frames `[type: u8][length: u32 BE][payload]`, encode/decode used by both holder and client, unit-testable without a PTY. Frame types: client→server `input` (raw bytes), `resize` (JSON `{cols, rows}`); server→client `snapshot`, `output`, `exit`. No JSONL/base64 — output is high-volume.
- Holder side: `tty.sock` serves the framed protocol: snapshot on connect (serialized from @xterm/headless via `@xterm/addon-serialize`, **including scrollback** — reviewing what happened is a primary reason to attach), then raw PTY output. Multiple simultaneous clients; the PTY size is the elementwise min over attached clients' sizes (see design-decisions.md "Attach semantics" — last-writer-wins was reverted after verification), applied as PTY resize (node-pty `resize()`, which delivers SIGWINCH only when the size actually changes). Snapshot correctness requirements (see work log for why): (a) a parse barrier — `terminal.write()` parses asynchronously, so buffer the new client's output relay until a write-callback barrier fires, then snapshot, then flush; (b) append the current cursor-visibility sequence (`?25l`/`?25h`) to the snapshot — the serialize addon does not capture it and pi runs cursor-hidden; (c) if pi's content reaches the bottom row, append a one-line scroll with the cursor re-parked one row higher, so the attach hint gets a row of its own below pi's content. On holder shutdown, send `exit` and destroy attach connections (don't just close the listener). A client that stops reading buffers unboundedly in Node — accepted for v1 (single machine, human attachers); revisit with a high-water disconnect if it bites.
- `pictl attach <agent>`: put the local terminal in raw mode, connect, send initial resize, render snapshot, proxy bytes both ways; forward local SIGWINCH as resize frames. Detach keybinding: `ctrl+]` (byte 0x1D; in raw mode received as a byte, not a signal), defined as a named constant in exactly one place — no magic strings — so it stays easy to change or make configurable later. After rendering the snapshot, draw a one-line hint ("attached to <id>; detach: ctrl+]") on the bottom row, bracketed by DECSC/DECRC. The terminal must be restored on every exit path: detach, agent death mid-attach (`exit` frame or socket close — report it), and unexpected errors.
- Attaching to a dormant agent is an error ("agent is dormant; run `pictl resume`") with a TODO comment: phase 3 replaces this with transparent revival.
- When the last attacher disconnects, the PTY keeps its last size.

Verification pause: spawn an agent, attach from two terminals simultaneously, interact with pi's TUI normally (including a `/` command), resize, detach/reattach and confirm the screen restores correctly, then kill the originating terminal and confirm the agent survives.

## Phase 3: RPC passthrough, tail, wait

Deliverables:

- One module (e.g. `src/rpc-commands.ts`) mapping CLI subcommands → RPC commands. This is the **only** file that should need editing when pi's RPC surface changes; keep everything else generic. Cover the **entire `RpcCommand` union** in pi's `rpc-types.ts` (decision 2026-06-12: full mirror; omissions need a very good reason): `prompt`, `steer`, `follow-up`, `abort`, `new-session`, `get-state`, `set-model`, `cycle-model`, `get-available-models`, `set-thinking-level`, `cycle-thinking-level`, `set-steering-mode`, `set-follow-up-mode`, `compact`, `set-auto-compaction`, `set-auto-retry`, `abort-retry`, `bash`, `abort-bash`, `get-session-stats`, `export-html`, `switch-session`, `fork`, `clone`, `get-fork-messages`, `get-entries`, `get-tree`, `navigate-tree`, `get-last-assistant-text`, `set-session-name`, `get-messages`, `get-commands`.
- Shared connection helper: connect to `<dir>/pi.sock`, validate the hello record, warn on pi version mismatch, send command, await the response, print result (`--raw` for raw response records; default prints response data).
- Agent addressing: positional `<agent-id>` accepting any unique prefix. Nothing else — session-id and workflow-role resolution were dropped (decision 2026-06-13): pictl has no concept of a "workflow", and two instances of the same workflow script would collide on a shared id anyway. Grouping is out-of-band instead: `spawn --tag <label>` records a non-unique label, and `list --cwd <dir>` (exact resolved path) plus `list --all` let a caller (e.g. a skill running `pictl list --cwd $(pwd)`) discover the agents sharing a directory along with their tags.
- Dormant-agent revival: every command that needs the socket transparently revives a dormant agent (equivalent to an implicit `pictl resume`) before proceeding. `list`/`status`/`gc` never revive. Revival must be mutually exclusive per agent: two commands reviving the same dormant agent concurrently would both `launchHolder`, and the holder's stale-socket cleanup would delete the first holder's _live_ pi.sock (see work log 2026-06-12 pre-phase-3). Serialize via e.g. an `O_EXCL` lock file in the agent dir; the loser waits for the winner's holder to become reachable instead of spawning.
- Holder fail-fast on the trust dialog (resolves open question 5): pi blocks on "Trust project folder?" before binding pi.sock when the cwd has `.pi` inputs and no stored trust decision, which would hang the headless holder forever. Rather than scan PTY output for the prompt text (decision 2026-06-13: string-scanning is fragile — it false-matches the phrase appearing in session history and cannot tell that `--approve` was passed), the holder reproduces pi's own decision deterministically before launching pi: `projectTrustWouldBlock(cwd, piArgs)` returns true iff no approve/deny flag is present, `hasProjectTrustInputs(cwd)`, the `ProjectTrustStore` has no stored decision for the cwd, and the default project trust is `ask` — using pi's exported `getAgentDir`/`hasProjectTrustInputs`/`ProjectTrustStore`/`SettingsManager`. On true it fails the spawn immediately with an error recommending `pictl spawn -- --approve`, before pi is started.
- `pictl tail <agent> [--follow] [--since <entry-id>] [--until <cond>] [--events]` — without `--follow`, dump entries from `get_entries` (since cursor if given) as JSONL and exit, printing the final entry id so callers can persist a cursor; with `--follow`, additionally stay connected and stream subsequent entries (use events as wakeups, re-drain `get_entries --since` to emit — never treat events as the data source). `--until <cond>` (added 2026-06-13) implies `--follow` and streams until a wait condition (below) holds, then drains trailing entries and exits 0 — the streaming counterpart of `wait`, reusing the same condition parser. Entry-id cursors are **session-scoped**: `/new`, `/resume`, `fork`, and `clone` replace the session, and `get_entries` errors on a `since` id not in the current session. Decision (2026-06-12): pass that error through as "entry not found; the session may have changed" and let the client decide what to do — pictl does not try to interweave session files. Clients that persist cursors should persist the sessionId alongside.
- `pictl wait <agent> --until turn-end|idle|no-activity:<secs> [--timeout <secs>]` — exit codes (decided 2026-06-12): 0 condition met, 1 agent dead or other runtime error, 2 usage error (bad arguments), 3 timeout. A dormant or archived agent has met any condition immediately (its process is doing nothing) and is **not** revived — reviving would only produce a guaranteed-idle agent. The shared `parseWaitCondition`/`applyWaitCondition` are reused by `tail --until` and `prompt --and-wait-until`. Semantics:
  - `turn-end`: return when the in-flight **or pending** turn ends — the next `agent_end` with `willRetry === false` (an `agent_end` with `willRetry: true` announces an auto-retry continuation, not a turn end). Returns immediately only if the agent is fully idle (`isStreaming` false AND empty pending queue). Treating an accepted-but-not-yet-started prompt as "a turn that must end" is what makes the sequential `pictl prompt X; pictl wait X --until turn-end` pattern race-free: `prompt` only exits after the RPC accepts the message, and the accepted-but-invisible window does not exist in pi (verified in source; see work log 2026-06-12 pre-phase-3).
  - `idle`: not streaming and pending queue drained (the common condition, so it gets the short name; renamed 2026-06-13 from `quiescent`). Known blind spots (auto-retry backoff, extension code) documented in the work log.
  - `no-activity:<secs>`: no session events for N seconds (fractional ok), regardless of streaming state (renamed 2026-06-13 from `idle:<secs>`). This intentionally differs from `idle`: it catches turns stalled mid-flight (e.g. blocked on human-facing extension UI, visible as `ui_wait_start` with no progress), which the supervisor pattern needs.
- `pictl prompt` ergonomics: message as positional argument, `-` to read from stdin (also on `steer`/`follow-up`); `--and-wait` blocks until the prompted turn ends (sugar for `--and-wait-until turn-end`), and `--and-wait-until <cond>` takes wait's full vocabulary — implemented on the single prompt connection (send prompt, await the condition on the same subscription) so it is race-free by construction and the recommended one-shot form. Busy behavior mirrors pi (decision 2026-06-12): if the agent is streaming, `prompt` errors unless `--streaming-behavior steer|follow-up` is given (mapped to the RPC `streamingBehavior` field) — deviating from pi's semantics would cause more confusion than benefit.
- Lifecycle commands (renamed/added 2026-06-13): `kill` became `purge` (the sole destructive command — delete the agent dir). New `archive` = `suspend` (wait for quiescence, stop the holder) plus an `archived` marker file, which hides the agent from `list` unless `--all`. Any resume — explicit `resume` or implicit revival via an RPC command — clears the marker. `suspend` stays distinct (dormant but still listed). The marker is a file in the agent dir rather than a field in holder-owned `agent.json`, mirroring `tombstone`, so it never races the holder's sole-writer ownership of the record.

Verification pause: drive a short conversation entirely from a second terminal via `pictl prompt`/`wait`/`tail --since`, including killing and re-running `tail` with a persisted cursor across a compaction (`pictl compact`) and confirming nothing is lost.

## Phase 4: supervisor/worker example + messaging skill

Deliverables:

- `examples/supervisor-worker.sh` — bash only, no SDK: create-or-resume two agents (persist their ids and a worker entry-cursor in a state dir), tell each the other's agent id, then loop: `wait` on the worker (turn-end or idle), drain new worker entries since cursor, format them, `prompt` the supervisor. Must be killable and re-runnable without losing its place (this is the acceptance test for the cursor design).
- A pi skill (markdown) teaching agents to use pictl: discovering self via `PI_AGENT_ID`, listing/addressing peers, `prompt`/`steer`/`interrupt`-via-`abort`, spawning sub-agents, and etiquette (prefer prompt over abort; do not kill agents you did not spawn).
- README updates documenting the end-to-end example.

Verification pause: run the example, attach to the supervisor, give it a task for the worker, watch the loop relay progress; kill and resume the orchestrating script mid-task.

### 2026-06-12: `confirmed` removed; review round 2 (commit a186630) addressed

- `SessionHistoryEntry.confirmed` removed entirely (user question: "so why do we have `confirmed` at all?"). It existed to predict, without touching the disk, whether a session file had been flushed — but once revival checked the disk directly, nothing behavioral read it; the dedup adoption logic only perpetuated the flag itself, and the status display is better served by current disk state than by a flag frozen at observation time. Removal deleted the holder's entire `message_end` handler and `currentSessionId` tracking; the event handler now reacts to `session_changed` alone. Old agent.json files with a `confirmed` field still parse (extra fields are ignored).
- Sequential awaits in `runStatus` (prefix resolution) and `forEachAgent` (load + action) parallelized via `Promise.all` — no ordering dependency existed, and concurrent actions keep quiescence waits from compounding across agents. Failures are still collected and reported in input order; per-agent output is single complete lines, so interleaving stays readable.
- Bug found and fixed: every suspend/kill of a running agent took exactly 5s. Not pi-side — pi handles SIGTERM as graceful shutdown (SIGINT is "abort turn") and exits in ~10ms. The cause was `terminatePi`'s un-cleared `Promise.race` escalation `setTimeout`: node exits on empty event loop _handles_ (it does not track promises), so the losing timer kept the CLI alive until it fired. Both race timers (`terminatePi`, `waitQuiescent` with `--timeout`) are now cleared after the race; suspend/kill of an idle agent measure ~0.1s, and `--timeout` expiry still reports "still busy" at the configured bound.
- Investigation side-note: SIGTERM landing while a connection sits un-accepted in pi's listen backlog leaves that client connection open with no close event. Irrelevant to current commands (all complete `hello` before signaling) but a hazard if a future command fire-and-forgets a connect.

### 2026-06-12: min-size resize policy; hint placement fix; review comment (commit a409b1e)

- **Environment**: the fork checkout is now on branch `anton/rpc-tree` (successor of `anton/pi-tee`: tee mode + `get_entries`/`get_tree` + `navigate_tree`, the last added 2026-06-12). `~/bin/pi` symlinks into that branch's dist, rebuilt the same day — installed binary and the `file:` type dependency are in sync. Branch references in both docs updated.
- **The `turn-end` race concern is closed — no accepted-but-invisible window exists.** Verified in pi source: the prompt RPC success response is emitted via `preflightResult(true)` (`src/core/agent-session.ts`, end of `prompt()`), and from there to `isStreaming = true` (`agent.ts` `runWithLifecycle`, reached through `agent.prompt` → `runPromptMessages`) the call chain is synchronous — no awaits. pi is single-threaded, so any `get_state` processed after the prompt response observes streaming-or-pending. Sequential `prompt; wait --until turn-end` is therefore race-free; `prompt --wait` remains pure ergonomics, not a correctness requirement.
- **`agent_end` semantics favor simple `wait` logic**: the agent loop drains both steering and follow-up queues before emitting `agent_end` (comment in `_handlePostAgentRun`), and retry/compaction/extension-queued continuations run as additional agent runs emitting their own `agent_end`s. The event carries `willRetry`. So "turn-end = next `agent_end` with `willRetry === false`" is well-defined, and `quiescent` differs only by messages third parties queue after that event.
- **Quiescence blind spot: auto-retry backoff** (affects existing `kill`/`suspend` too, not just `wait`). Between a retryable failure and the retry, pi sleeps with exponential backoff (`_prepareRetry` in `agent-session.ts`) while `isStreaming` is false and `pendingMessageCount` is 0. A connected waiter can skip `agent_end` events with `willRetry: true`, but a client connecting mid-backoff cannot tell — `get_state` exposes no retry state. Decision: accept and document; no pi-side change now (limiting divergence from upstream outweighs the gap — the turn in question already errored). Recommendation recorded for later: add `retryPending: boolean` to `RpcSessionState` pi-side and upstream it.
- **Quiescence blind spot: extension code.** There is no general "extension code running" indicator. What does exist: (a) a `prompt` invoking an extension command gets its RPC response only after the handler returns (`_tryExecuteExtensionCommand` is awaited before `preflightResult` in `prompt()`), so `pictl prompt "/cmd"` blocks for the handler's duration and a subsequent `wait` returning immediately is correct — the agent is not working; (b) extension UI waits emit `ui_wait_start`/`ui_wait_end` side-channel events (`src/modes/interactive/interactive-mode.ts` `withUiWaitEvent`), which is what makes `idle:<secs>` catch UI-stalled turns. Invisible: extension event-hook code and background work a handler starts without awaiting — `kill`/`suspend`/`wait` can observe "quiescent" mid-extension. Accepted for v1.
- **Entry cursors are session-scoped** — `/new`, `/resume`, `fork`, `clone` replace the session (each broadcasts `session_changed`), and `get_entries` errors on a `since` id not in the current session. Decision: `tail` passes the error through ("entry not found; the session may have changed"); recovery is client policy, and pictl does not interweave session files. Cursor-persisting clients (phase-4 example) should persist sessionId alongside the entry id.
- **Concurrent transparent-revival race** (new hazard once revival is implicit; explicit `resume` never raced in practice): two commands reviving the same dormant agent both `launchHolder`, and the holder unconditionally removes existing sockets before spawning pi — the second holder would delete the first's _live_ pi.sock and spawn a second pi. ### 2026-06-12: phase 3 step 5 — `pictl tail` (awaiting check-in)

- `src/tail.ts`: entries as JSONL, one per line, with a final `{"type":"pictl_cursor","sessionId":...,"entryId":...}` record (the plan's "print the final entry id" deliverable, shaped as a JSONL record so `tail -1 | jq` extracts it and the sessionId travels with it, per the cursors-are-session-scoped decision). A cursor record follows every drain batch in `--follow` mode, so a killed-and-rerun consumer resumes from its last persisted line.
- The cursor is the last _printed entry's_ id (file order), deliberately not the response's `leafId`: `navigate-tree` can move the leaf backwards onto an already-printed entry, and a leafId cursor would then re-emit everything after it.
- `--follow` per the decided design: events are pure wakeups (any event except the per-token `message_update`/`tool_execution_update` noise — unknown future event types default to waking), every drain re-issues `get_entries --since <cursor>`. **Session replacement mid-follow resyncs quietly**: the cursor is repositioned to the new session's tip without dumping its history (only entries created after the replacement stream out), announced by a cursor record with the new sessionId.
